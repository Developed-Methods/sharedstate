🗺 1. Executive Overview

sharedstate is a Rust library that synchronizes a user-defined deterministic state machine across a cluster using gossip-based leader election and TCP streaming replication. It is a well-conceived system with a solid test suite — but its implementation has grown organically until roughly 80% of the critical logic lives inside a single 1,736-line file (node.rs), with multiple panics replacing proper error paths and several undocumented invariants.

The goal of the rewrite is not to change what the library does — the protocol, the election algorithm, and the public trait surface all remain. The goal is to make the implementation legible: each file has one job, every error is handled explicitly, and every invariant is encoded in the type system rather than asserted at runtime.

Current largest file
1,736 lines — src/shared/node.rs — contains peer observation, election application, follow-connection management, action forwarding, incoming-peer dispatch, and subscription streaming all interleaved.
Target after rewrite
<300 lines per file. Every file has a single stated responsibility. No file contains more than one async worker type.
Panic count (current)
6 reachable panic! calls in production paths — subscribe retries, broadcast-offline, state-worker lag, and message encoding assertions.
Panic count (target)
0 panics in production paths. Every current panic is replaced with a typed Result or a graceful shutdown signal.

⚠ 2. Current Pain Points

ID Location Severity Description
P1 node.rs (whole file) Critical 1,736-line god-file. Peer observation, election, follow management, action forwarding, incoming-connection dispatch, and subscription streaming are all interleaved with no boundary between concerns.
P2 node.rs:391 Bug Action pump uses hardcoded sleep(500ms) instead of NodeTiming::follow_retry_interval. Tests and deployments cannot tune it. Documented in plan.txt as M3.

P10 recoverable_state.rs Low The 2048-entry history cap is hard-coded with no warning mechanism when a slow follower falls outside the window and will require a full state dump on reconnect.

📐 3. Rewrite Principles

One file, one job
Every source file encapsulates exactly one async worker, one data structure family, or one pure algorithm. No file exceeds 350 lines. If it does, it splits.
Errors, not panics
Every failure path that currently panic!s is replaced with a typed Result. Workers propagate errors to a NodeSupervisor that decides whether to restart or shut down.
Invariants in types
If a condition must be true (e.g., "the broadcast sender is always alive while a node is leader"), encode it as a type that cannot be constructed unless the condition holds, rather than asserting it at runtime.
Lock granularity
Split ControlState's single coarse lock into three finer-grained structures — peer map, election state, and follow connection — so slow operations (peer I/O) cannot block fast ones (election tick).
Preserve the public API
The DeterministicState trait, NodeState, NodeTiming, and NodeActionSender signatures stay. The M3 changes (SendActionError::NoLeader, generic A param on sender) are included as the only breaking changes.
No behaviour change
Election algorithm, wire protocol, recovery protocol, and test scenarios are not altered. The existing integration test suite and recorded-replay scenarios are the acceptance criteria for the rewrite.

🏗 4. New Architecture

The rewrite reorganises the implementation into five clear layers:

LayerModuleResponsibility
State state/ User-facing DeterministicState trait; RecoverableState wrapper; lock-free SharedState / SharedStateHandle (unchanged from current)
Protocol protocol/ All wire message types (SyncRequest, SyncResponse) and the binary framing layer. Isolated so the message set can evolve independently.
Transport transport/ Abstract SyncIO / SyncIOAddress traits; ReadChannel / WriteChannel async workers; SimulatedNet. No cluster logic here.
Cluster cluster/ All cluster logic split across focused submodules: leader state, follower connection, peer observer, peer handler, action pump, election. Described in detail below.
Testing testing/ SharedStateTestOrchestrator, recording/replay infrastructure. Currently lives at crate root — moved to its own module for clarity.

Worker supervision model

Currently workers (StateMaintainWorker, the client loop, the peer handler, the action pump) are spawned as fire-and-forget tasks. When one fails, the node silently degrades. The rewrite introduces a lightweight NodeSupervisor that owns all worker JoinHandles and a shared CancellationToken. When any worker returns an error, the supervisor cancels remaining workers and reports the failure via a watch::Receiver<NodeStatus> so callers can observe node health. 

pub enum NodeStatus {
    Running,
    Degraded { reason: String },   // one non-critical worker failed, node still operates
    Stopped,                           // clean shutdown requested
    Failed { reason: String },        // critical worker failed, node must be restarted
}

🧩 6. Key Type Redesigns

6a. Split ControlState into three narrow locks

The current single Mutex<ControlState> means a slow peer-observation network call blocks the election tick. Split into three independently locked structures:

Before (node.rs)
struct Inner<A, D> {
    control: Mutex<ControlState<A, D>>,
    authorative: Mutex<AuthorativeState<D>>,
    // ...
}

struct ControlState<A, D> {
    leader_info: LeaderInfo<A>,
    peers: HashMap<A, Option<PeerDetails<A>>>,
    follow: Option<FollowConnection<A, D>>,
    election: ElectionState<A>,
}
After (cluster/control.rs)
struct NodeShared<A, D> {
    // fine-grained: each lock held briefly
    peers:    RwLock<PeerMap<A>>,
    election: Mutex<ElectionState<A>>,
    follow:   Mutex<FollowSlot<A, D>>,
    leader:   Mutex<LeaderSlot<D>>,
    // channels, no lock needed
    leader_tx: watch::Sender<LeaderInfoMessage<A>>,
    actions_tx: mpsc::Sender<D::Action>,
}

Key insight: PeerMap is read by observation workers and written only during election. Using RwLock allows concurrent reads. ElectionState and FollowSlot are only touched by the single client-worker task — their Mutex is uncontended, just for Sync.

6b. AuthorativeState — replace panics with Results

Before (authorative_state.rs)
pub fn subscribe(&self) -> (...) {
    for _ in 0..16 {
        // try to align snapshot + rx head
        if let Some(result) = self.try_subscribe() {
            return result;
        }
        std::thread::sleep(Duration::from_millis(1));
    }
    panic!("failed to subscribe");
}

pub fn apply_authority(&self, action: ...) {
    self.broadcast.send(action)
        .unwrap_or_else(|_| panic!("authority broadcast is offline"));
}
After (cluster/leader.rs)
pub async fn subscribe(
    &self
) -> Result<..., LeaderError> {
    tokio::time::timeout(
        Duration::from_millis(50),
        self.try_subscribe_loop()
    ).await
    .map_err(|_| LeaderError::SubscribeTimeout)?
}

pub fn apply_authority(
    &self, action: ...
) -> Result<(), LeaderError> {
    self.broadcast.send(action)
        .map_err(|_| LeaderError::BroadcastOffline)
}

6c. StateMaintainWorker — propagate lag to supervisor

Currently the worker silently breaks on Lagged, leaving SharedState frozen. In the rewrite it returns a typed error that the supervisor can act on:

cluster/leader.rs — StateMaintainWorker::run
- Err(SequencedRecvError::Lagged) => { - tracing::error!("state maintain worker lagged, giving up"); - break; // SharedState silently freezes - } + Err(SequencedRecvError::Lagged) => { + return Err(WorkerError::StateMaintainLagged); + // supervisor receives this, cancels all workers, + // and sets NodeStatus::Failed { reason: "state worker lagged" } + }

6d. NodeActionSender — M3 fix (NoLeader + timing)

This is the fix described verbatim in plan.txt, incorporated into the rewrite:

Before
pub enum SendActionError {
    Closed,
}

pub struct NodeActionSender<Action> {
    tx: mpsc::Sender<Action>,
}

pub async fn send(&self, action: Action)
    -> Result<(), SendActionError>
{
    self.tx.send(action).await
        .map_err(|_| SendActionError::Closed)
}

// action_pump (node.rs:391)
tokio::time::sleep(
    Duration::from_millis(500) // hardcoded
).await;
After (cluster/action_pump.rs)
pub enum SendActionError {
    Closed,
    NoLeader,   // new: fast-fail during leaderless period
}

pub struct NodeActionSender<Action, A: SyncIOAddress> {
    tx: mpsc::Sender<Action>,
    leader: watch::Receiver<LeaderInfoMessage<A>>,
}

pub async fn send(&self, action: Action)
    -> Result<(), SendActionError>
{
    if self.leader.borrow().leader.is_none() {
        return Err(SendActionError::NoLeader);
    }
    self.tx.send(action).await
        .map_err(|_| SendActionError::Closed)
}

// action_pump uses NodeTiming
tokio::time::sleep(
    timing.follow_retry_interval  // was 500ms hardcoded
).await;

6e. Fix SyncRequest wire ID gaps

Current IDs skip 2 and 5. The rewrite introduces a compatibility decode layer so existing peers using the old IDs continue to work, while new code uses clean sequential IDs: 

// protocol/messages.rs
// New canonical IDs (0-12 sequential). read_from() accepts both old and new IDs.
fn id_to_variant(id: u8) -> Result<SyncRequestKind, io::Error> {
    match id {
        0  => Ok(SyncRequestKind::ProtocolVersion),
        1  => Ok(SyncRequestKind::MyAddress),
        // 2 was removed in an old version — accept it as unknown gracefully
        2  => Err(unknown_id_err(2)),
        3  | 13 => Ok(SyncRequestKind::Ping),  // 13 = canonical
        // ... etc
    }
}

🧪 9. Testing Strategy

Existing test suite as acceptance criteria

The current library has an unusually strong test foundation. These tests are the acceptance criteria for the rewrite — all must pass at the end of every phase:

Unit tests (pure functions)
8 election algorithm tests, 7 message framing tests, 5 client policy tests, 7 follow policy tests, path validation tests. These are the easiest to preserve — they test pure functions that are not moving.
Integration tests (SimulatedNet)
12 named scenario tests in node/test.rs covering leader election, failover, path propagation, and reconnect behaviour. These are the primary regression guard for Phase 6.
Recorded-scenario replay tests
3 JSON recordings (new-leader-indirect, leader-return, leader-return2) replayed in test_orchestrator/test.rs. These guard edge-case convergence behaviour from real observed failures.
Fuzzy stabilization test
fuzzy_cluster_stabilization_smoke (seeded, 5 s, 3 + 3 nodes) runs in CI. The long variant (300 s) should be run manually before each phase merge.
