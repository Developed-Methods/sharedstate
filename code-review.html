The two most severe findings — the quorum calculation and the leader check TOCTOU — can produce split-brain state corruption in any partitioned deployment. For a 20-datacenter system these must be addressed before production use. All other findings are independently actionable.

Architecture Overview

Core Layers

  • DeterministicState — user-defined CRDT/state machine
  • RecoverableState — wraps user state with generation IDs & history for follower recovery
  • AuthorativeState — leader-side sequenced broadcast to followers
  • SharedState / HotRead — lock-free snapshot reads for local consumers

Consensus Model

Leader-election gossip: every node periodically pings peers, collects ElectionObservations, and calls decide_election to converge on a leader. Non-leaders establish a long-lived follow connection to stream authority actions. The protocol resembles a simplified Raft without persistent log or formal quorum writes.

Network

  • Custom length-prefixed framing (u32 header)
  • Keepalive (0) and close (0xFFFFFFFF) sentinel frames
  • Separate short-lived probe connections (observe) vs long-lived follow connections
  • Peer discovery gossip via SharePeers exchange

Key Invariants

  • Only the leader calls apply_authority
  • Every authority action carries a monotone sequence number
  • Generation ID changes on every leadership promotion to allow follower recovery detection
  • Leader paths must be cycle-free and end at the local node

Findings

  • C1 Quorum computed from reachable nodes — minority partition can elect a leader (split-brain)
  • C2 TOCTOU between leader check and apply_authority — non-leader can corrupt state
  • H1 Follow reader unconditionally take()s current follow — races with new connection, loses it
  • H2 Peer-controlled vector length → unbounded Vec::with_capacity → OOM / DoS
  • M1 subscribe() panics after 16 retries — crashes the node task
  • M2 apply_authority panics on closed broadcast — hard crash instead of graceful shutdown
  • M3 Action pump silently starves during leaderless intervals — actions are dropped after 1024 queue depth
  • L1 RecovGenerationEnd::STATIC_SIZE undercounts by one field
  • L2 apply_update panics with a truncated message — obscures root cause

C1 — Quorum computed from reachable nodes: minority partition can elect a leader

src/shared/node/election.rs : 159–163
Critical

What the code does. After tallying which leader-capable nodes are reachable, decide_election computes a majority threshold and decides whether to promote the local node: 

let reachable_count = input.known_can_lead.iter()
    .filter(|addr| reachable_within_stale(addr, ...))
    .count();
let active_can_lead_count = reachable_count.max(usize::from(input.can_lead));
let majority = active_can_lead_count / 2 + 1;         // ← bug: denominator is reachable, not total

if leader == input.local_address && input.can_lead && majority <= reachable_count {
    return ElectionDecision::PromoteSelf { observed_term: term };
}

The denominator for the majority threshold is derived from reachable_count, not from the total known_can_lead.len(). This means "majority" shrinks whenever nodes go unreachable — exactly the moment correctness matters most.

Concrete split-brain scenario — 5-node cluster, 3+2 partition

Cluster: nodes 1–5, all can_lead = true, known_can_lead = {1,2,3,4,5}.

Partition A (nodes 1, 2, 3 see each other):
reachable_count = 3 → majority = 3/2+1 = 2 → 2 ≤ 3 → node 1 promotes ✓

Partition B (nodes 4, 5 see each other):
reachable_count = 2 → majority = 2/2+1 = 2 → 2 ≤ 2 → node 4 promotes ✗

Both sides elect a leader. Both leaders accept writes. State permanently diverges.

Fix

Use the total cluster size as the denominator, not the reachable subset:

let total_can_lead = input.known_can_lead.len().max(usize::from(input.can_lead));
let majority = total_can_lead / 2 + 1;

if leader == input.local_address && input.can_lead && majority <= reachable_count {
    return ElectionDecision::PromoteSelf { ... };
}

With this fix: Partition B has reachable_count = 2 < majority = 3 → does not promote.

C2 — TOCTOU between leader check and apply_authority: non-leader can corrupt state

src/shared/node.rs : 356–366  |  1582–1601
Critical

In the local action pump (start_local_action_pump), the node checks whether it is the current leader, releases the control mutex, and then calls apply_authority under a separate state mutex lock: 

// lock acquired and released here:
let is_leader = inner.control.lock().await.leader.leader == Some(inner.address);
if is_leader {
    let authority = {
        let state = state_handle.read();           // reads state snapshot
        state.authority(RecoverableStateAction::StateAction { action })
    };
    // ← leadership can change here; control lock not held
    inner.state.lock().await.apply_authority(authority).await;  // writes!
    break;
}

Between the leadership check and apply_authority, Tokio can schedule another task that: (a) strips this node of leadership, and (b) lets a new leader start accepting writes. When this node resumes, it applies an authority action it should not — resulting in a state sequence that no follower will receive from the new leader.

The same pattern appears in handle_action (line 1582–1601), where an incoming peer action is applied: a peer sends an action to what it believes is the leader; the receiving node checks is_leader, releases the control lock, and applies the action.

Failure scenario

Node A is leader. Node A and Node B simultaneously detect a network partition. Node B promotes itself (C1 aside). Node A's action pump task is scheduled, checks is_leader = true, yields, B's promotion is recorded, then A's pump resumes and calls apply_authority. A's followers receive an action that B (the new leader) never issued, permanently diverging from B's state.

Fix

Re-check leadership under the same lock that guards AuthorativeState, or hold a single combined lock across the check and the apply. Alternatively, gate apply_authority on an epoch/term token that is invalidated on leader change: before applying, verify the term has not advanced since the check.

H1 — Follow reader unconditionally take()s current follow: races with new connection

src/shared/node.rs : 1266–1277
High

When a follow reader task exits (stream closed, cancelled, etc.) it cleans up: 

// Step 1: unconditionally take whatever follow is currently set
let closed_follow = { inner.control.lock().await.follow.take() };
// ← lock is released; another task can now set a NEW follow connection

if let Some(follow) = closed_follow {
    if follow.remote == target {
        // correct path: cancels old follow, clears leader
        follow.cancel.cancel();
        inner.clear_remote_leader().await;
    } else {
        // wrong: tries to put back whatever we took — but a third connection
        // may have been established in the gap between take() and here
        let _ = inner.control.lock().await.follow.replace(follow);
    }
}

If the follow connection was replaced by the time the reader exits (because the leader changed and connect_follow established a new one), the reader take()s the new connection. The else branch then tries to put it back, but the control lock was released between take() and replace(). A third connection established in that window is silently overwritten and lost.

The node ends up holding a stale follow reference that points to a dead channel, causing it to never reconnect until the next full observation cycle detects the anomaly.

Fix

Only take the follow if it matches the exiting reader's target. Hold the lock across any subsequent state mutations:

let mut control = inner.control.lock().await;
if control.follow.as_ref().is_some_and(|f| f.remote == target) {
    if let Some(follow) = control.follow.take() {
        follow.cancel.cancel();
        if let Some(Some(details)) = control.peers.get_mut(&target) {
            details.connected = details.active_connections > 0;
        }
        drop(control);
        inner.clear_remote_leader().await;
    }
}
// else: a different follow is active; leave it untouched

H2 — Peer-controlled vector length causes unbounded allocation: OOM / DoS

src/shared/messages.rs : 377  |  src/state/recoverable_state.rs : 230
High

Two deserialization sites trust a peer-supplied u64 as a vector length and immediately allocate capacity: 

// messages.rs — read_vec (called for SharePeers, LeaderPath, reachable_can_lead, etc.)
fn read_vec<T: MessageEncoding, R: std::io::Read>(read: &mut R) -> std::io::Result<Vec<T>> {
    let count = u64::read_from(read)? as usize;
    let mut vec = Vec::with_capacity(count);   // ← allocates count * sizeof(T) bytes
    for _ in 0..count { ... }
    Ok(vec)
}

// recoverable_state.rs — RecoverableStateDetails::read_from
let len = u64::read_from(read)? as usize;
let mut history = VecDeque::with_capacity(len); // ← same issue

A rogue or compromised peer sends a message with count = 2^32. For a 32-byte element type, Vec::with_capacity attempts to allocate 16 GiB. The allocator either OOMs and aborts the process, or on systems with overcommit, the subsequent element writes trigger the OS killer. Either way the node crashes.

RecoverableStateDetails history is internally capped at 2048 on write (line recoverable_state.rs:78) but this cap is not enforced on deserialization, so a peer claiming 2 billion history entries is accepted.

Fix

Cap allocations at a sane maximum before allocating capacity. For history, use the existing write-side cap:

// read_vec
const MAX_VEC_ITEMS: usize = 65_536;
let count = u64::read_from(read)? as usize;
if count > MAX_VEC_ITEMS {
    return Err(std::io::Error::new(std::io::ErrorKind::InvalidData, "vector too large"));
}
let mut vec = Vec::with_capacity(count);

// recoverable_state.rs history
const MAX_HISTORY: usize = 2048;
if len > MAX_HISTORY {
    return Err(std::io::Error::new(std::io::ErrorKind::InvalidData, "history too large"));
}

M1 — subscribe() panics after 16 retries: crashes the node task

src/shared/authorative_state.rs : 74–93
Medium

AuthorativeState::subscribe retries up to 16 times (spaced 1ms apart) to obtain a broadcast subscription aligned with the current state snapshot. On the 17th failure it panics: 

for _ in 0..16 {
    let state_borrow = handle.read();
    let Ok(sub) = self.authority_broadcast.subscribe_from(state_borrow.accept_seq()).await else {
        handle.quiescent();
        tokio::time::sleep(Duration::from_millis(1)).await;
        continue;
    };
    return (state.clone(), sub);
}
panic!("failed to subscribe");  // ← kills the task and all its children

subscribe is called from PeerWorker::subscribe_fresh and subscribe_recovery, which are invoked on every follower connection. Under high write throughput the broadcast ring can race the state snapshot repeatedly for longer than 16ms, triggering the panic. The panic unwinds PeerWorker, kills its spawned feed task, and may leave the follower stuck with no state stream.

Fix

Return an error instead of panicking, and let the caller close the connection gracefully:

pub async fn subscribe(&self) -> Option<(RecoverableState<D>, ...)> {
    for _ in 0..32 {
        ...
        let Ok(sub) = ... else { ...; continue; };
        return Some((state, sub));
    }
    tracing::error!("failed to subscribe after retries");
    None
}

M2 — apply_authority panics on closed broadcast: hard crash instead of graceful shutdown

src/shared/authorative_state.rs : 137–143
Medium

If the internal broadcast channel is closed for any reason (bug in StateMaintainWorker, task crash, or a logical error during reset), every subsequent call to apply_authority panics: 

pub async fn apply_authority(&mut self, authority: ...) {
    let success = self.authority_tx.send(authority).await.is_ok();
    if !success {
        panic!("authority broadcast is offline");
    }
}

This turns any internal transient failure into a hard node crash. For a system expected to run across 20 datacenters indefinitely, crashes from internal errors should be logged and trigger graceful re-initialization, not process abort.

Fix

Return a Result from apply_authority and propagate the error to the caller for clean shutdown or recovery:

pub async fn apply_authority(&mut self, authority: ...) -> Result<(), AuthBroadcastClosed> {
    self.authority_tx.send(authority).await.map_err(|_| AuthBroadcastClosed)
}

M3 — Action pump silently starves during leaderless intervals: actions drop after queue fills

src/shared/node.rs : 354–394
Medium

When neither a local leader nor a follow connection exists, the action pump spins with a 500ms sleep per attempt, holding each pending action captive: 

while let Some(mut action) = rx.recv().await {
    loop {
        let is_leader = ...;
        if is_leader { ... break; }

        let follow_tx = ...;
        if let Some(follow_tx) = follow_tx {
            match follow_tx.send(...).await {
                Ok(()) => break,
                Err(e) => { action = recover(e); }   // re-enter loop
            }
        }
        tokio::time::sleep(Duration::from_millis(500)).await;  // blocks next action
    }
}

The local_actions_tx channel is bounded at 1024 items (line 177). During an election (which can take multiple observation intervals at 3s each), 1024 user-submitted actions fill the queue. Further action_sender().send(...) calls return Err(SendActionError::Closed) — misleadingly named; the channel is full, not closed — and the actions are silently discarded. Callers have no way to distinguish "leader elected, please retry" from "node is shutting down".

Fix

Add a distinct SendActionError::NoLeader variant, or expose backpressure. The 500ms hardcoded sleep should also respect the configurable NodeTiming values rather than being a magic constant.

L1 — RecovGenerationEnd::STATIC_SIZE undercounts by one field

src/state/recoverable_state.rs : 189–191
Low / Latent

RecovGenerationEnd has three u64 fields (old_id, generation, next_sequence), but its STATIC_SIZE constant only sums two: 

impl MessageEncoding for RecovGenerationEnd {
    const STATIC_SIZE: Option<usize> = m_opt_sum(&[u64::STATIC_SIZE, u64::STATIC_SIZE]);
    //                                                ^ two u64 = 16 bytes
    // write_to writes three u64 fields = 24 bytes
    fn write_to<T: Write>(&self, out: &mut T) -> std::io::Result<usize> {
        let mut sum = 0;
        sum += self.old_id.write_to(out)?;        // 8
        sum += self.generation.write_to(out)?;     // 8
        sum += self.next_sequence.write_to(out)?;  // 8
        Ok(sum)  // = 24
    }
}

Currently harmless because RecovGenerationEnd is only serialized as part of RecoverableStateDetails, never as a top-level framed message. The mismatch would trigger the assert_eq!(size, bytes_written) assertion in send_message if this type were ever framed directly, producing a confusing panic instead of a compile-time error.

Fix
const STATIC_SIZE: Option<usize> = m_opt_sum(&[u64::STATIC_SIZE, u64::STATIC_SIZE, u64::STATIC_SIZE]);

L2 — apply_update panics with a truncated message: obscures root cause

src/state/shared_state.rs : 107–109
Low / Latent

The state update function panics with an incomplete string literal if the sequence invariant is violated: 

fn apply_update(&mut self, update: &Self::Action) {
    match update {
        HotSharedStateUpdate::Authority { seq, action } => {
            if self.state.accept_seq() != *seq {
                panic!("Received authority ")   // ← message is cut off mid-sentence
            }
            assert_eq!(self.state.accept_seq(), *seq);   // ← redundant: already checked above
            self.state.update(action);
            assert_eq!(self.state.accept_seq(), seq + 1);
        }
        ...
    }
}

The panic message appears to be a copy-paste artifact left mid-edit. More critically, it provides no information about the expected vs. actual sequence, making post-mortem analysis from logs nearly impossible. The immediately following assert_eq!(self.state.accept_seq(), *seq) is also dead code — the panic above would have already fired.

Fix
if self.state.accept_seq() != *seq {
    panic!(
        "authority action sequence mismatch: expected {}, got {}",
        self.state.accept_seq(), seq
    );
}
// remove the redundant assert_eq below

Recommendations

Immediate (before any production traffic)

  • Fix the quorum denominator (C1) — cannot be deferred for multi-DC use
  • Fix the follow-reader take/replace race (H1)
  • Add vector length caps on deserialization (H2)
  • Address the leader TOCTOU (C2) — requires a design decision on locking strategy

Short-term

  • Replace panics in subscribe and apply_authority with returned errors (M1, M2)
  • Expose a NoLeader error variant from the action sender (M3)
  • Fix the truncated panic message (L2)
  • Fix STATIC_SIZE on RecovGenerationEnd (L1)

Testing gaps

  • No tests for partition scenarios — add simulation tests with test_orchestrator that split and re-join nodes
  • No fuzz tests for deserialization — run cargo-fuzz on SyncRequest::read_from
  • The fuzzy.rs test file exists but is not integrated into CI (not listed in lib.rs)

Design considerations

  • The protocol has no persistent log — node crash loses all in-flight actions since last snapshot
  • Leader path relay (chain of follow connections) introduces cascading failure modes not currently handled
  • Consider a term-based authority token threaded through apply_authority to close the TOCTOU fundamentally
